Accessibility Essentials Guide

Accessibility isn't a feature you bolt on at the end — it's about making sure everyone can use what you build. People with visual impairments, motor disabilities, cognitive differences, or temporary injuries (a broken arm, bright sunlight on a screen) all benefit from accessible design. This guide shows you what Vanduo handles for you, and what you need to do yourself.

Why This Matters

1 in 4 Adults

have some form of disability. That's 25% of your potential users.

It Helps Everyone

Captions help in noisy rooms. High contrast helps in sunlight. Keyboard nav helps power users.

It's Often Required

Laws like ADA, Section 508, and EAA mandate accessibility for many websites.

The right mindset: Don't think of accessibility as "extra work for edge cases." Think of it as building something that actually works for everyone — the same way a ramp helps wheelchairs, strollers, delivery carts, and tired hikers.

Essential 1 Keyboard Navigation

Many users navigate with keyboards alone — Tab to move between elements, Enter/Space to activate, Escape to close. Here's how to ensure your site works without a mouse:

Tab Order

By default, browsers tab through elements in DOM order. If your visual layout differs from the source order (e.g., using CSS Grid reordering), users may get confused. Use tabindex sparingly and only when necessary:

<!-- DON'T: Manually managing tab order gets messy fast -->
<button tabindex="3">Third</button>
<button tabindex="1">First</button>
<button tabindex="2">Second</button>

<!-- DO: Let the natural DOM order be the tab order -->
<button>First</button>
<button>Second</button>
<button>Third</button>

Skip-to-Content Link

Help keyboard users skip past the navbar to the main content:

<!-- First element in <body> -->
<a href="#main-content" class="vd-skip-link">Skip to main content</a>

<nav class="vd-navbar">...</nav>

<main id="main-content">
    <!-- Your page content -->
</main>
Pro tip: Test your entire site using only the keyboard. Unplug your mouse (or just don't touch it) and try to complete every user action. You'll immediately find problems — elements you can't reach, invisible focus states, trapped focus.

Essential 2 Screen Reader Essentials

Screen readers convert your page into speech or Braille. They rely on HTML structure and ARIA attributes to convey meaning. Here's what you need to add yourself:

Label Everything

Without Labels

<!-- Screen reader says: "Button" -->
<button>
    <i class="ph ph-x"></i>
</button>

<!-- Screen reader says: "Image" -->
<img src="hero.jpg">

<!-- Screen reader says: "Edit text" -->
<input type="text">

Users have no idea what these elements do.

With Labels

<!-- "Close dialog" -->
<button aria-label="Close dialog">
    <i class="ph ph-x"></i>
</button>

<!-- "Sunset over the ocean" -->
<img src="hero.jpg" alt="Sunset over the ocean">

<!-- "Email address, edit text" -->
<label for="email">Email address</label>
<input type="text" id="email">

Every element communicates its purpose.

Use Semantic Elements

<!-- DON'T: Divs with click handlers -->
<div onclick="doSomething()">Click me</div>

<!-- DO: Use the right element -->
<button onclick="doSomething()">Click me</button>

<!-- Buttons are focusable, announce as "button", and respond to
     Enter/Space automatically. Divs do none of that. -->

Essential 3 Color and Contrast

Text needs enough contrast against its background to be readable. WCAG requires:

  • AA (minimum): 4.5:1 ratio for normal text, 3:1 for large text (18px+ bold or 24px+)
  • AAA (enhanced): 7:1 ratio for normal text, 4.5:1 for large text
Watch out: When you change colors using the Theme Customizer, you might accidentally create low-contrast combinations. Always check that your text remains readable after customization — especially with lighter primary colors like amber or lime on white backgrounds.

Never rely on color alone to convey information:

Color Only

<!-- Red text means error... but
     colorblind users can't tell -->
<p style="color: red;">
    Invalid email
</p>

Color + Icon + Text

<!-- Clear even without color -->
<div class="vd-alert vd-alert-danger">
    <i class="ph ph-warning-circle"></i>
    <strong>Error:</strong>
    Please enter a valid email address.
</div>

Essential 4 Forms Accessibility

Forms are where accessibility creates or breaks the experience. Every input field needs a proper label, and errors need to be communicated clearly:

<!-- Accessible form field with error -->
<div class="vd-form-group">
    <label for="user-email" class="vd-form-label">Email Address</label>
    <input type="email"
           id="user-email"
           class="vd-form-control"
           aria-describedby="email-error"
           aria-invalid="true"
           placeholder="you@example.com">
    <p id="email-error" class="vd-form-error" role="alert">
        <i class="ph ph-warning-circle"></i>
        Please enter a valid email address.
    </p>
</div>

Key patterns:

  • for="id" on <label> connects it to the input — clicking the label focuses the input
  • aria-describedby connects the input to its error message — screen readers read both
  • aria-invalid="true" tells screen readers this field has an error
  • role="alert" on the error message makes screen readers announce it immediately when it appears
Pro tip: Never use placeholder as a replacement for <label>. Placeholders disappear when users start typing, leaving them wondering what they're supposed to enter. Always use a visible label.

Essential 5 Testing Your Work

You don't need to be an accessibility expert to test your site. Here are the most effective approaches, from easiest to most thorough:

1. Keyboard Test

Don't touch your mouse. Tab through every page. Can you reach everything? Can you see where focus is? Can you activate buttons with Enter?

Time: 5 minutes per page

2. Browser DevTools

Chrome's Lighthouse (Audits tab) has an Accessibility score. Run it and aim for 90+. It catches missing labels, low contrast, and missing ARIA.

Time: 1 minute per page

3. axe DevTools Extension

A free browser extension that finds accessibility issues and explains how to fix them. Much more detailed than Lighthouse.

Time: 2-3 minutes per page

4. Screen Reader

Try VoiceOver (macOS: Cmd+F5), NVDA (Windows, free), or TalkBack (Android). Listen to your site — does it make sense?

Time: 10-15 minutes (first time)

Pro tip: Vanduo's own documentation site uses Playwright for automated a11y testing. If you have Playwright set up, you can use @axe-core/playwright to catch accessibility regressions in CI.

Accessibility Checklist

Bookmark this quick-reference checklist for every page you build:

Structure

  • Page has one <h1> and headings don't skip levels (h1 → h2 → h3)
  • Page uses <main>, <nav>, <header>, <footer> landmarks
  • Page has a <title> that describes the content
  • Language is set: <html lang="en">

Images & Media

  • All <img> tags have alt text (or alt="" for decorative images)
  • Icons used as buttons have aria-label
  • Videos have captions or transcripts

Interaction

  • All interactive elements are reachable via keyboard (Tab / Shift+Tab)
  • Focus is visible on all focusable elements
  • Clickable elements use <button> or <a>, not <div>
  • Escape closes modals, dropdowns, and overlays

Forms

  • Every input has a visible <label>
  • Required fields are indicated (not just with color)
  • Error messages use aria-describedby and role="alert"

Color & Visual

  • Text contrast is at least 4.5:1 (check with DevTools)
  • Information isn't conveyed by color alone
  • Site is usable when zoomed to 200%
