ESM vs IIFE vs Legacy Guide

JavaScript module formats can feel confusing — ESM, IIFE, CJS, UMD… what do these acronyms even mean? This guide breaks it all down in plain language, helps you pick the right one for your project, and shows you exactly how to set each up with Vanduo.

The Big Picture

When you include JavaScript on a web page, the browser needs to know how that code is packaged. Think of module formats like different types of packaging for the same product:

IIFE

Immediately Invoked Function Expression

The classic <script> tag approach. Drop it in, it just works. Creates a global Vanduo object.

ESM

ECMAScript Modules

The modern standard. Use import / export. Works with bundlers like Vite, Webpack, Rollup.

CJS (Legacy)

CommonJS

The Node.js format. Uses require(). For older build tools or server-side rendering.

Which One Should I Choose?

Follow this decision tree to find the right format for your project:

Are you using a bundler? (Vite, Webpack, Rollup, Parcel, esbuild)

Yes → Use ESM

You get tree-shaking (smaller bundles), modern syntax, and the best developer experience.

No → Next question...

Do you need to support very old browsers? (IE11, old Edge)

Yes → Use CJS with a bundler/transpiler

No → Use IIFE

Pro tip: If you're just getting started or building a simple site, IIFE is your best friend. No build tools, no configuration — just a <script> tag and you're done. You can always switch to ESM later when you adopt a bundler.

Format 1 IIFE — The Drop-In Approach

Best for: Static sites, prototypes, learning, WordPress themes, simple projects without a build step.

<!-- Add these before </body> -->
<script src="https://cdn.jsdelivr.net/gh/Nostromo-618/vanduo-framework@main/dist/vanduo.min.js"></script>
<script>
    // Vanduo is now available as a global object
    Vanduo.init();

    // Use any component
    Vanduo.components.toast.show({
        title: 'Hello!',
        message: 'IIFE is working.',
        type: 'success'
    });
</script>

How it works: The browser downloads the script, executes it immediately, and attaches Vanduo to the global window object. Every component is included — you don't pick and choose.

Pros:

  • Zero configuration
  • Works everywhere
  • No build tools needed
  • Great for CDN delivery

Cons:

  • Loads the entire library (no tree-shaking)
  • One global variable (potential naming conflicts)
  • No static analysis from your editor

Format 2 ESM — The Modern Standard

Best for: Projects using Vite, Webpack, Rollup, or any modern bundler. SPAs, React/Vue/Svelte apps, production sites where bundle size matters.

// In your main JavaScript file (e.g., main.js or app.js)
import { Vanduo } from './dist/vanduo.esm.js';

// Or if installed via npm:
// import { Vanduo } from 'vanduo-framework';

// Initialize
Vanduo.init();

// Use components with full IntelliSense support
Vanduo.components.toast.show({
    title: 'Hello!',
    message: 'ESM is working.',
    type: 'success'
});

To use ESM directly in a browser (without a bundler), you need type="module":

<script type="module">
    import { Vanduo } from './dist/vanduo.esm.js';
    Vanduo.init();
</script>
Watch out: type="module" scripts are deferred by default and have their own scope — you can't access variables from a module script in a regular script or inline onclick handlers. If you need global access, stick with IIFE.

Pros:

  • Tree-shaking (smaller bundles)
  • Better editor IntelliSense
  • Static analysis and linting
  • The web platform standard

Cons:

  • Needs a bundler for best results
  • Slightly more setup
  • Module scope can surprise beginners

Format 3 CJS — The Legacy Path

Best for: Node.js environments, server-side rendering (SSR), older bundlers that don't support ESM, legacy codebases being gradually modernized.

// CommonJS / require() syntax
const { Vanduo } = require('vanduo-framework');

// Or point directly to the CJS file:
const { Vanduo } = require('./dist/vanduo.cjs.js');

Vanduo.init();
Watch out: CJS does not work in browsers without a bundler. If you're building a client-side website, use IIFE or ESM instead. CJS is primarily for server-side code and build pipelines.

Pros:

  • Works with Node.js natively
  • Compatible with older tooling
  • Synchronous loading

Cons:

  • No tree-shaking
  • Not browser-native
  • Being superseded by ESM in Node.js

Before / After: Loading Components

Here's the real-world difference — loading Vanduo with each format when you only need toasts and modals:

IIFE / CJS

Full bundle 
<!-- Loads ALL components -->
<script src="vanduo.min.js"></script>
<script>
    Vanduo.init();
    // You get toast + modal + every
    // other component whether you
    // use them or not
</script>

Loads: ~45 KB gzipped (entire framework)

ESM with Bundler

Tree-shaken 
// Only imports what you use
import { Vanduo } from 'vanduo-framework';

Vanduo.init();
// Your bundler strips out unused
// components automatically
// (tree-shaking)

Loads: Only the code you actually use

Still not sure? Start with IIFE. It works without any tooling, and you can always migrate to ESM later when your project grows. See the Module Formats reference in Concepts & Architecture for the full technical details.

For npm Users: Package Exports

If you install Vanduo via npm, the package.json exports field automatically gives you the right format:

# Install
npm install vanduo-framework

# In your code:
import { Vanduo } from 'vanduo-framework';  // → gets ESM
const { Vanduo } = require('vanduo-framework'); // → gets CJS

Your bundler reads the "exports" field in package.json and selects the correct file automatically. You don't need to specify the file path.
